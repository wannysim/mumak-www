# Expressive Refactor Guide

이 스킬은 "주석은 변명이다"라는 철학을 바탕으로, 코드를 '설명'하는 대신 '표현'하게 만듭니다. 다만 이 저장소에서는 과한 추상화나 멋을 위한 패턴보다, **읽히고 유지보수되며 테스트 가능한 코드**를 우선합니다.

## 1. 핵심 철학: Expressive Code

- **주석 최소화**: 로직을 설명하는 주석은 코드의 실패를 의미합니다. 주석을 달기 전, 함수명과 변수명을 개선하여 의도를 드러내십시오.
- **문맥(Context)의 격리**: 복잡한 로직은 읽는 이에게 노이즈가 됩니다. 캡슐화를 통해 필요한 정보만 노출하십시오.
- **선언적 지향**: '어떻게(How)' 하는지 절차를 나열하지 말고, '무엇(What)'을 하려는지 결과 중심으로 작성하십시오.
- **검증 가능성 유지**: 표현력이 높아져도 테스트가 어려워지면 좋은 리팩토링이 아닙니다. 작은 함수, 예측 가능한 입력/출력, 얕은 의존성을 유지하십시오.

## 저장소 특화 우선순위

이 저장소는 React/Next.js/Turborepo 모노레포입니다. 따라서 리팩토링 시 우선순위는 아래 순서를 따릅니다.

1. **이름 개선** - 변수, 함수, 컴포넌트 이름이 의도를 드러내게 만듭니다.
2. **함수 추출** - 큰 조건식, 파생 계산, 렌더링 분기를 작은 함수로 나눕니다.
3. **파일/모듈 분리** - 함께 변하는 코드끼리 모으고, 거대한 JSX 블록이나 복잡한 유틸을 분리합니다.
4. **조건식/데이터 흐름 정리** - predicate extraction, early return, map/filter/reduce 전환을 검토합니다.
5. **IIFE/체이닝 적용** - 위 단계로도 충분하지 않을 때만 제한적으로 사용합니다.

## 2. 기술적 지침 (Technical Guidelines)

### A. 클로저 & IIFE를 통한 스코프 오염 방지

특정 계산이나 초기화에만 쓰이는 임시 변수들은 외부 스코프를 오염시키지 않도록 즉시 실행 함수(IIFE)나 클로저 내부에 가둘 수 있습니다.

다만 다음 경우에는 IIFE보다 **이름 있는 함수 추출**을 우선합니다.

- 재사용 가능성이 있는 계산
- 테스트에서 직접 검증하고 싶은 계산
- React 컴포넌트 본문이 이미 복잡한 경우
- Server Component / data fetching 로직처럼 흐름 파악이 중요한 경우

```javascript
// Good: IIFE를 통한 의미 단위 캡슐화
const finalPrice = (() => {
  const basePrice = 10000;
  const tax = 1.1;
  const discount = user.isMember ? 0.1 : 0;
  return basePrice * (1 - discount) * tax;
})();
```

### B. 유려한 메서드 체이닝 (Fluent Interface)

비즈니스 로직의 연속성을 표현할 때는 메서드 체이닝이 도움이 될 수 있습니다.

하지만 이 저장소에서는 아래와 같은 경우 **억지로 체이닝하지 않습니다**.

- React 컴포넌트 내부 상태 변경
- 비동기 흐름과 예외 처리
- 타입 추론이 급격히 복잡해지는 유틸
- 단순 함수 나열로도 충분히 읽히는 코드

체이닝이 유효한 경우:

- 도메인 오브젝트의 연속된 조작
- 설정 builder
- 절차보다 흐름이 중요하고, 각 단계 이름이 명확한 경우

### C. 서술적 조건식 (Predicate Extraction)

복잡한 논리 연산자는 그 자체로 암호가 됩니다. 비즈니스 언어를 담은 변수나 함수로 추출하십시오.

- `if (user.isActive && user.lastLogin > lastWeek)` (X)
- `if (user.isRecentActiveUser())` (O)

### D. React/Next 리팩토링 기본기

- 거대한 JSX 블록은 작은 컴포넌트로 분리합니다.
- `useEffect`는 표현력 개선을 위해 늘리지 않습니다.
- props drilling보다 무조건 context를 넣지 말고, 먼저 컴포넌트 경계를 조정합니다.
- 서버/클라이언트 경계가 흐려지지 않게 `use client` 필요성을 재확인합니다.
- 상대경로를 늘리기보다 저장소의 절대경로 import 규칙을 유지합니다.

### E. 금지 사례

- "멋있어 보여서" 메서드 체이닝을 새로 도입
- 테스트 가능한 순수 함수를 IIFE 안으로 숨기기
- 작은 파일 여러 개를 억지로 합쳐 거대한 추상화 계층 만들기
- 주석을 지우면서 이름은 그대로 두기
- 표현력을 이유로 타입 안정성을 해치기
- 리팩토링 후 검증 없이 마무리하기

## 3. 리팩토링 수행 단계 (Execution Steps)

1. **동작 보존 범위 확인**: 이번 변경이 기능 추가인지, 순수 리팩토링인지 먼저 구분합니다.
2. **이름 개선**: 변수/함수/컴포넌트 이름부터 고쳐 의도를 드러냅니다.
3. **구조 정리**: 함수 추출, early return, JSX 분리, 모듈 경계 재조정으로 복잡도를 낮춥니다.
4. **스코프 정리**: 상위 스코프의 임시 변수들을 별도 함수, 클로저, IIFE로 격리합니다.
5. **선언형 전환**: 반복과 조건문을 필요할 때만 더 읽기 쉬운 선언형 형태로 바꿉니다.
6. **패턴 적용**: 정말 도움이 되는 경우에만 IIFE나 체이닝을 도입합니다.
7. **검증**: 관련 테스트 및 최소한의 검증 명령을 직접 실행합니다.

## 4. 리팩토링 후 검증

- 변경한 모듈의 테스트가 있으면 우선 실행합니다.
- 테스트가 없더라도 최소한 `check-types`, `lint`, 관련 테스트 여부를 검토합니다.
- UI나 라우팅을 건드렸다면 E2E 필요성도 판단합니다.
- 순수 리팩토링이라고 해도 회귀 가능성을 항상 의심합니다.

## 5. 최종 체크리스트 (Quality Gate)

- [ ] 코드가 영어 문장처럼 막힘없이 읽히는가?
- [ ] 불필요한 임시 변수가 완전히 제거되거나 격리되었는가?
- [ ] "왜 이렇게 짰지?"라는 의문이 드는 지점에 주석 대신 더 명확한 이름이 쓰였는가?
- [ ] 로직의 '호흡'이 중간에 끊기지 않고 유려하게 이어지는가?
- [ ] 메서드 체이닝/IIFE가 "멋"이 아니라 실제 가독성 향상에 기여하는가?
- [ ] 테스트 가능성과 타입 안정성이 오히려 좋아졌는가?
- [ ] 이번 리팩토링이 저장소의 import/component 패턴과 충돌하지 않는가?
