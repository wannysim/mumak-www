요즘 나는 Cursor IDE를 단순한 "AI가 코드 대신 쳐주는 편집기"로 보지 않는다.

오히려 **여러 층의 제어 장치와 실행 도구를 가진 작업 환경**에 가깝다고 느낀다.

처음엔 그냥 채팅과 자동완성만 썼다.
하지만 조금씩 더 써보니, 그 위에 규칙을 얹고, 스킬을 가르치고, 서브에이전트에게 일을 나누고, 훅으로 위험한 동작을 막고, MCP로 외부 세계를 연결할 수 있었다.

이 문서는 그런 기능들을 한 번에 정리해두기 위한 노트다.
완성된 사용 설명서라기보다는, **내가 현재 Cursor를 어떤 레이어로 이해하고 있는지** 기록하는 쪽에 가깝다.

## 먼저 큰 그림

내 기준으로는 대략 이렇게 나뉜다.

- **Plugins** - 이 모든 것을 묶어서 배포하는 패키지
- **Rules** - AI가 계속 참고할 기준
- **Skills** - 특정 작업을 수행하는 방법을 가르친 실행형 지식
- **Subagents** - 메인 에이전트가 위임하는 보조 작업자
- **Hooks** - 에이전트 루프의 특정 지점에 끼어드는 제어 장치
- **MCP** - 외부 도구와 데이터를 연결하는 통로

즉, Cursor는 단순히 "모델 하나"로 움직이는 게 아니라,
**지침, 워크플로우, 위임, 제어, 외부 연결**이 겹쳐진 구조로 이해하는 편이 더 정확하다고 본다.

## 1. Plugins

[공식 문서](https://cursor.com/docs/plugins)

플러그인은 Rule, Skill, Agent, Command, MCP 서버, Hook 등을 한 번에 묶어 배포하는 단위다.

그래서 이 기능을 개별적으로 이해하는 것도 중요하지만, 결국 팀 단위로 보면 Plugins가 가장 바깥 레이어가 된다.
개인 설정을 넘어, **"우리 팀은 AI를 이런 방식으로 쓰겠다"**를 패키지처럼 배포할 수 있기 때문이다.

이 지점이 흥미롭다.
보통 IDE 플러그인이라고 하면 문법 하이라이팅이나 Lint 확장 정도를 떠올리는데,
Cursor의 플러그인은 **AI가 어떤 문맥을 받고, 어떤 도구를 쓰고, 어떤 규칙을 따르는지**까지 함께 전달한다.

문서에서 특히 인상적이었던 점은 두 가지다.

- 플러그인은 Git 저장소 기반으로 배포된다.
- 마켓플레이스 플러그인은 오픈소스이며 수동 리뷰를 거친다.

즉, 편의성만이 아니라 **투명성과 검토 가능성**도 같이 가져가려는 방향으로 보인다.

내 기준에서 Plugins는 "매일 직접 만지는 기능"이라기보다,
`Rules + Skills + MCP + Hooks`를 팀 단위로 묶어 배포하는 **상위 배포 단위**에 가깝다.

## 2. Rules

[공식 문서](https://cursor.com/docs/rules)

Rules는 에이전트에게 주입되는 지속적인 기준이다.

쉽게 말하면 매번 채팅할 때마다
"우리는 이런 구조를 선호하고, 이런 컨벤션을 따르고, 이런 파일은 이렇게 다뤄라"
를 계속 말해주는 셈이다.

이 프로젝트만 봐도 이미 그런 흔적이 많다.

- import 경로 규칙
- 컴포넌트 export 패턴
- 테스트 파일 네이밍
- Git/PR 작성 방식
- 데이터베이스 접근 정책

이런 건 사람이 매번 프롬프트로 설명하면 금방 빠뜨리게 된다.
그래서 Rules는 결국 **AI에게 팀의 기본기와 습관을 주입하는 층**이라고 보는 게 맞다.

문서상 Rules는 몇 가지 층위로 나뉜다.

- Project Rules - 저장소 안 `.cursor/rules`
- User Rules - 내 전역 설정
- Team Rules - 조직 단위 규칙
- `AGENTS.md` - 좀 더 단순한 형태의 지침 파일

이 차이도 중요하다.
모든 걸 한 군데에 몰아넣기보다, **어디까지가 프로젝트 컨벤션이고 어디까지가 개인 취향인지**를 나눌 수 있기 때문이다.

내가 느끼기엔 Rules는 "AI를 똑똑하게 만드는 기능"이라기보다,
**AI가 자꾸 같은 실수를 반복하지 않게 만드는 기능**에 더 가깝다.

## 3. Skills

[공식 문서](https://cursor.com/docs/skills)

Skills는 특정 작업을 수행하는 방법을 에이전트에게 가르치는 패키지다.

여기서 중요한 건 단순 프롬프트가 아니라는 점이다.
문서 기준으로 Skills는 `SKILL.md`뿐 아니라 `scripts/`, `references/`, `assets/`까지 포함할 수 있다.

즉, "이럴 때 이런 식으로 생각해" 수준을 넘어서,
**"이 작업을 할 땐 이 문서를 참고하고, 이 스크립트를 실행하고, 이 순서로 처리해"**까지 담을 수 있다.

그래서 나는 Rules와 Skills를 이렇게 구분해서 이해하고 있다.

- **Rules** - 항상 지켜야 하는 기준
- **Skills** - 특정 작업을 수행하는 방법

예를 들면:

- Jira 티켓을 기준으로 개발 시작하기
- changelog 생성하기
- Turborepo에서 영향 범위만 점검하기
- 특정 배포 절차를 따라가기

이런 것들은 컨벤션이라기보다 **반복 가능한 워크플로우**에 가깝다.
그래서 Rules보다 Skills 쪽에 더 잘 어울린다.

문서에서 마음에 들었던 부분은, Skills가 필요할 때만 점진적으로 로드된다는 점이다.
이런 설계는 결국 컨텍스트 창을 아끼려는 방향이기도 하다.

요즘 AI 도구를 쓰다 보면 "모든 걸 프롬프트에 다 때려넣는 방식"이 금방 한계에 부딪히는데,
Skills는 그걸 파일 구조와 실행 흐름으로 분리하려는 시도처럼 보인다.

## 4. Subagents

[공식 문서](https://cursor.com/docs/subagents)

Subagents는 메인 에이전트가 작업을 위임하는 보조 작업자다.

내가 이 개념을 중요하게 보는 이유는 성능보다도 **컨텍스트 분리** 때문이다.
코드베이스를 뒤지는 긴 탐색, 장황한 터미널 출력, 브라우저 제어 결과 같은 것들은 메인 대화창을 금방 오염시킨다.

Subagents는 그런 무거운 작업을 바깥에서 처리하고,
메인 에이전트에게는 결과만 요약해서 돌려준다.

이건 실제로 꽤 큰 차이를 만든다.

- 코드베이스 전체 탐색
- 긴 테스트 실패 로그 분석
- 구현 후 검증 전용 에이전트 분리
- 서로 다른 작업을 병렬로 진행

문서에서도 `Explore`, `Bash`, `Browser` 같은 빌트인 서브에이전트를 따로 두는 이유를 설명하는데,
결국 공통점은 하나다.
**중간 산출물이 너무 시끄럽다**는 것이다.

그래서 내 기준에서 Subagents는 "더 똑똑한 비서"라기보다,
**맥락을 분리해서 메인 작업 흐름을 지키는 장치**에 가깝다.

Rules와 Skills가 "무엇을 기준으로, 어떻게 할 것인가"를 다룬다면,
Subagents는 "누가 이 일을 맡을 것인가"를 다룬다.

## 5. Hooks

[공식 문서](https://cursor.com/docs/hooks)

Hooks는 내가 보기엔 가장 강력하면서도, 가장 조심해서 다뤄야 하는 기능이다.

이건 AI에게 능력을 더 주는 기능이 아니라,
**AI가 움직이는 루프 사이사이에 사람이 개입할 지점을 만드는 기능**이기 때문이다.

문서상 Hooks는 여러 단계에서 동작할 수 있다.

- 세션 시작 전후
- 프롬프트 제출 전
- 파일 읽기 전 / 파일 수정 후
- Shell 실행 전후
- MCP 실행 전후
- Subagent 시작/종료 시점

즉, 에이전트의 행동을 그냥 지켜보는 게 아니라,
필요하면 막고, 수정하고, 추가 문맥을 넣고, 자동 후처리를 붙일 수 있다.

이게 중요한 이유는 명확하다.

- 위험한 Shell 명령 차단
- SQL write 같은 민감 동작 통제
- 비밀 키나 개인정보 유출 검사
- 파일 저장 후 포매터 자동 실행
- 특정 MCP 호출에 승인 절차 추가

결국 Hooks는 **자율성을 높이는 기능**이면서 동시에 **자율성을 제어하는 기능**이기도 하다.

AI를 더 많이 믿기 위해서는,
오히려 이런 식의 제약과 감시 장치가 더 필요하다고 생각한다.

## 6. MCP

[공식 문서](https://cursor.com/docs/mcp)

MCP는 Cursor를 외부 세계와 연결하는 표준 프로토콜이다.

파일 몇 개 읽는 수준을 넘어서,
Jira, Figma, GitHub, 데이터베이스, 브라우저, 배포 플랫폼 같은 외부 시스템과 연결할 수 있게 해준다.

이 지점부터는 Cursor가 단순 IDE를 넘어,
**내 로컬 코드베이스를 중심으로 여러 도구를 묶는 작업 허브**처럼 느껴진다.

문서에서 제공하는 예시 서버만 봐도 범위가 넓다.

- GitHub
- Figma
- Atlassian
- Playwright
- DB/클라우드/관측 도구

중요한 건 MCP가 단순 API 호출 모음이 아니라는 점이다.
문서 기준으로 MCP 서버는 `Tools`, `Prompts`, `Resources` 등을 노출할 수 있다.
즉, 에이전트는 외부 시스템을 그냥 "참고"하는 수준이 아니라, **읽고, 질문하고, 실행하는 인터페이스**로 다룰 수 있다.

내 작업 흐름에 대입하면 이런 식이다.

- Jira 티켓을 읽고 작업 맥락 파악
- Figma를 보고 UI 구현
- GitHub 상태 확인
- DB 스키마 조회
- Playwright나 브라우저 계열 도구로 실제 동작 검증

그래서 MCP는 부가 기능이 아니라,
**AI의 컨텍스트를 코드 바깥으로 확장하는 핵심 레이어**라고 느낀다.

## 그럼 무엇을 언제 써야 할까

여기서부터는 내 식의 구분이다.

- 프로젝트 전반에 계속 적용되어야 하면 `Rules`
- 특정 작업 절차를 반복 가능하게 만들고 싶으면 `Skills`
- 긴 탐색이나 검증을 메인 흐름과 분리하고 싶으면 `Subagents`
- 위험한 행동을 통제하거나 자동 후처리를 붙이고 싶으면 `Hooks`
- 외부 도구와 실제로 연결해야 하면 `MCP`
- 이 모든 묶음을 팀에 배포하고 싶으면 `Plugins`

이렇게 보면 각각의 역할이 꽤 분명해진다.

예를 들어 "Jira를 읽고 우리 팀 규칙대로 구현하고, 완료 후 테스트를 검증한다"는 흐름을 만든다고 해보자.

- Jira 연결은 `MCP`
- 팀 코딩 기준은 `Rules`
- 티켓 기반 개발 절차는 `Skills`
- 검증 전용 보조 작업자는 `Subagents`
- 위험한 명령 차단이나 후처리는 `Hooks`
- 이 전체 세트를 팀에 뿌리는 건 `Plugins`

결국 이 기능들은 서로 경쟁하지 않는다.
오히려 **서로 다른 층에서 맞물린다.**

## 지금의 결론

아직 나도 이 기능들을 전부 깊게 써본 것은 아니다.
특히 Hooks와 Plugins는 "할 수 있는 것"에 비해 아직 충분히 체화되진 않았다.

그럼에도 지금은 한 가지를 분명히 느낀다.

Cursor의 핵심은 모델 하나의 성능이 아니라,
**그 모델을 어떤 문맥 위에 올리고, 어떤 규칙으로 묶고, 어떤 도구와 연결하고, 어디서 제어할 것인가**에 있다.

AI 도구를 잘 쓴다는 건 결국 좋은 프롬프트를 한 번 던지는 일이 아니라,
이런 레이어들을 조금씩 정리해서 **내 작업 환경 자체를 설계하는 일**에 더 가까운 것 같다.

이 노트도 그 설계의 초안이다.

관련해서는 [[pkm|PKM]], [[what-is-digital-garden|디지털 가든이란 무엇인가]], 그리고 [내가 AI시대에서 살아남는 방법](/ko/blog/essay/ai-survival)도 이어서 참고할 수 있다.
